# OPSEC Rules — что можно и что нельзя в этот репо

**OPSEC** = operational security, оперативная безопасность. Правила того, что можно публиковать в этот репо, а что — нельзя.

> Этот репо **публичный**. Всё, что попадает в commit, становится доступно всему интернету навсегда (даже если потом удалить файл — он остаётся в git-истории). Поэтому каждое сообщение, каждый файл, каждый commit message проходит через эти правила.

---

## Принцип

> **Если есть сомнение — публикуй в приватной зоне.**

Приватные зоны (где можно класть что угодно):
- Локальная папка `C:\Dev\Codex plus Cloud\` на машине Геннадия
- Приватный репо `transkrib-knowledge` (внутренняя база знаний)
- Чаты с Cloud / Codex напрямую

Публичная зона (этот репо) — только то, что прошло санитизацию.

## ❌ НИКОГДА не публиковать

### Секреты и доступы

- API-ключи, токены, пароли — **в любом виде**, включая частичные, маскированные, «бывшие» (старые тоже могут пригодиться атакующему как stepping stone)
- OAuth/SSO-токены, JWT, сессионные cookies
- SSH-ключи, GPG-ключи
- Пароли от баз данных, S3-credentials, webhook-secrets
- Содержимое любых `.env` файлов и переменных окружения

### Внутренние идентификаторы инфраструктуры

- Service IDs Railway (вида `eb3eb8ff-155e-...`)
- Project IDs Railway (вида `c8e80613-...`)
- Render service IDs (вида `srv-d6voti95...`)
- Environment IDs, deployment IDs
- Supabase Project URLs и Anon Keys
- Cloudflare R2 bucket IDs

> Почему: эти ID позволяют атакующему сразу обратиться к конкретному ресурсу. Даже без ключа — это разведданные.

### Финансовые данные

- Конкретные суммы выручки, балансы, расходы
- ShopID платёжных систем (ЮKassa и аналоги)
- Внутренние идентификаторы платёжных провайдеров
- Реальные суммы тарифов с указанием продавца «у нас»

> Названия типа «Базовый/Стандарт/Годовой» и публичные цены на нашем сайте — это публично всегда, их можно. А «выручка 0 за 365 дней» — нет.

### Активные уязвимости

- Детальные описания уязвимостей в коде, который сейчас работает в проде
- Эксплуатационные подробности (как именно атаковать)
- Bug #26 (`SECRET_KEY` офлайн-валидация лицензий) — НЕ публикуем
- Bug #28 (Electron `fs:readFile` без валидации) — НЕ публикуем
- Любая будущая уязвимость подобного класса — оценивается отдельно

> Bug #27 (LemonSqueezy webhook не подключён) — **можно**, потому что это архитектурный недостаток, без эксплоит-информации. Чтобы атаковать — надо быть пользователем-плательщиком, и максимум, что получится — не активировать самому себе тариф (что вредит только самому себе).

### Персональные данные

- Имена, email, телефоны клиентов
- Telegram ID, GitHub username других людей (своими можно)
- Список бета-тестеров с контактами
- Любая информация, которая может быть приписана к конкретному человеку

### Внутренние процессы и слабости

- Список текущих не закрытых проблем с привязкой к конкретным точкам атаки
- Информация о том, что какой-то сервис временно «спит» или работает с ограничениями
- Расписания деплоев, окна обслуживания (полезно атакующему для timing-атак)

## ⚠️ Проверочный чеклист перед commit

Перед тем как добавить файл в этот репо:

1. **Открой файл и прочитай его полностью.**
2. Пройди по списку выше — есть ли в файле хоть один пункт из «никогда»?
3. Если есть — **в эту зону файл не идёт**. Положи в приватную папку или приватный репо.
4. Если сомневаешься — пингани остальных участников команды через приватный канал, не пуш.
5. После публикации проверь raw-URL файла на github.com — действительно ли видно то, что хотел показать.

## Что делать, если секрет случайно попал в репо

1. **Немедленно остановиться** — не делать новых коммитов, не двигаться дальше.
2. **Считать секрет скомпрометированным.** Ротировать его в источнике (сменить пароль в кабинете, перегенерировать API-ключ, и т.д.) — потому что репо публичный, и секрет уже мог быть просканирован ботами GitHub-секрет-сканеров и распространён.
3. После ротации — `git filter-repo` для удаления секрета из истории (или просто пометить, что значение теперь мёртвое).
4. Записать инцидент в `DECISIONS.md` с разбором, как именно сорвалось и что изменить в процессе.

> **Удалить файл `git rm` НЕДОСТАТОЧНО.** Файл остаётся в истории git. Единственный надёжный путь — считать секрет утечённым и ротировать.

## Изменения в этих правилах

Эти правила могут эволюционировать. Любое изменение — через коммит в этот же репо с явным указанием в commit message: `chore(opsec): update rules — <причина>`. История изменений правил видна в git log этого файла.
